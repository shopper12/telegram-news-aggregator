# App briefing pipeline with Make, OpenAI, Notion, and Stock Scanner app

## Goal

Make the market briefing visible inside the Android app without relying on ChatGPT conversation automations to push data externally.

The correct data flow is:

```text
Make scheduled scenario
  -> market data fetch / news fetch
  -> OpenAI structured briefing generation
  -> Notion database item for audit/history
  -> POST to Render API /api/recommendations
  -> Android Stock Scanner app reads /api/recommendations
  -> Bot recommendation screen shows the cards
```

ChatGPT recurring tasks in the ChatGPT app are not the storage layer. They can still be used for reading a briefing in the conversation, but the app-visible pipeline must be generated by Make or another external scheduler.

## Existing repo support

This repository already exposes the recommendation bridge.

- GET /api/recommendations returns the latest stored recommendation payload.
- POST /api/recommendations accepts a recommendation payload and stores it locally.
- GET /api/recommendations.txt returns a compact plain-text version for messenger commands.
- Authentication uses X-API-Key and the CHAT_PICKS_API_KEY or NEWS_BOT_API_KEY environment variable.

The Android Stock Scanner app already reads:

```text
https://stock-scanner-api-5sk6.onrender.com/api/recommendations
```

Therefore the minimum working structure is Make -> POST /api/recommendations -> Android app refresh.

## Make scenario

### 1. Trigger

Use Make Scheduler.

Recommended times in Asia/Seoul:

```text
09:05 KST  Korea-open morning briefing
16:05 KST  Korea post-close plus U.S. preview
```

### 2. Market data collection

Use HTTP modules or market-data providers to collect:

```text
KOSPI, KOSDAQ, KOSPI200
S&P500, Nasdaq100 or Nasdaq Composite, Dow, Russell2000
Nikkei, Hang Seng, TAIEX, Euro Stoxx 50 when relevant
U.S. 2Y and 10Y yields
DXY, USD/KRW, WTI, Brent, gold, VIX, Fear and Greed when available
BTC spot from two sources
sector ETF / relative strength snapshot
news 3 to 5 items
earnings calendar
```

Do not let the OpenAI step invent prices. Pass verified values into the prompt.

### 3. OpenAI generation

Use an OpenAI module or HTTP request to OpenAI API.

The OpenAI output must be structured and must contain exactly 3 final recommendations. Defensive or no-trade can be one of the 3.

Required top-level fields:

```yaml
briefing_datetime_kst: YYYY-MM-DD HH:mm
mode: morning | post_close | weekend_holiday
source: make_openai_market_briefing
summary: short Korean summary
market_status: current market status in Korean
recommendations:
  - asset_name:
    ticker:
    market: KR | US | CRYPTO | ETF | CASH
    direction: long | short | inverse_long | defensive | no_trade
    basis_price:
    basis_price_currency: KRW | USD | USDT
    basis_timestamp_kst:
    basis_source:
    entry:
    stop:
    target1:
    target2:
    invalidation:
    holding_period:
    reason:
    risk:
    strategy_rule_applied:
performance_review:
missed_opportunity:
strategy_evolution:
strategy_ledger:
briefing_markdown:
```

The current Android card screen displays the recommendation fields. The optional briefing_markdown field is for future full-briefing display or Notion storage.

### 4. Notion audit log

Create one Notion database item per briefing.

Recommended properties:

```text
Briefing
Briefing Datetime KST
Mode
Source
Recommendation Count
Ticker 1
Ticker 2
Ticker 3
Status
Error
```

The Notion step is only an audit/history layer. The Android app must not read Notion directly because a Notion token must not be embedded in the app.

### 5. POST to Render API

Use Make HTTP module.

Method:

```text
POST
```

URL:

```text
https://stock-scanner-api-5sk6.onrender.com/api/recommendations
```

Headers:

```text
Content-Type: application/json
X-API-Key: value of CHAT_PICKS_API_KEY in Render
```

Body:

Use the structured OpenAI output from step 3.

### 6. Android app display

The app opens the Bot recommendation screen and fetches the latest payload from /api/recommendations.

The user-visible sequence is:

```text
Open Stock Scanner app
Tap bot recommendation / market briefing menu
App calls /api/recommendations
Cards render the latest 3 recommendations
```

## Local test

Create a payload file and post it manually.

```bash
curl -X POST https://stock-scanner-api-5sk6.onrender.com/api/recommendations \
  -H 'Content-Type: application/json' \
  -H 'X-API-Key: YOUR_CHAT_PICKS_API_KEY' \
  --data-binary @payload.json
```

Then verify:

```bash
curl https://stock-scanner-api-5sk6.onrender.com/api/recommendations
```

If the response contains the same 3 tickers, the app can read the same data.

## Required Render environment variables

```env
CHAT_PICKS_API_KEY=strong-secret-value
CHAT_PICKS_PATH=/tmp/chat_picks.json
```

Optional remote source mode:

```env
CHAT_PICKS_SOURCE_URL=
STOCK_SCANNER_RECOMMENDATIONS_URL=
```

Leave remote source mode empty when Make directly POSTs into this API.

## Failure rules

- If Make fails before OpenAI, do not update Notion or API.
- If OpenAI output has fewer or more than 3 recommendations, stop the scenario.
- If any recommendation lacks basis price, timestamp, or source, stop the scenario or replace it with no-trade.
- If Notion succeeds but API POST fails, the app will not update. Mark Notion Status as error.
- If API POST succeeds but Notion fails, the app can still update. Mark audit failure separately.

## Implementation decision

Use Notion as audit/history, not as the app data source.
Use Render /api/recommendations as the app data source.
Use Make/OpenAI as the generator and scheduler.
Do not rely on ChatGPT conversation automations for app-visible data.
